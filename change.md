# Current release summary

**Version:** 4.4.0  
**Date:** 2026-04-18

## What changed

- **v4.4.0 — Sandbox-only dev login (`/devlogin`).** Adds a dedicated page that lets reviewers click into the user and admin sides without a password, gated by **four independent safety layers** so it can never reach production. New `POST /api/auth/dev-login` (body `{ role: "user" | "admin" }`) issues a short-lived local-session JWT for a **synthetic** identity (`local:devlogin-user` — no roles, or `local:devlogin-admin` with `comp-plan-admin`) — the identities are **not inserted into `local_users`**, so they don't interfere with "last admin" safeguards or normal sign-in. Every call writes an `auth_audit` row (`action = "dev_login_used"`). **Gate 1 (build):** new `npm run build:sandbox` loads `.env.sandbox` which sets `VITE_DEV_LOGIN_ENABLED=true`; the standard `npm run build` leaves the flag undefined so the login buttons never render. **Gate 2 (server):** `registerDevLoginRoutes()` is a no-op unless `ENABLE_DEV_LOGIN=true` — the POST route simply doesn't exist otherwise (404). **Gate 3 (startup):** `assertDevLoginSafeForStartup()` throws during boot if `ENABLE_DEV_LOGIN=true` under `NODE_ENV=production` without the explicit `CONFIRM_DEV_LOGIN_IN_PRODUCTION=yes-i-really-want-this` override, so the API refuses to start. **Gate 4 (audit):** every use is recorded. New `GET /api/auth/dev-login/status` feeds the SPA. New page `src/pages/DevLoginPage.tsx` at `/devlogin` — two buttons (Dev User Login / Dev Admin Login), hard-navigates to `/app` or `/admin/` after signing in so both SPAs reinitialise from storage; renders a safety stub when either gate is off. `scripts\push-to-sandbox.ps1` gains a `-WithDevLogin` switch that builds via `build:sandbox` and prints a reminder to also set `ENABLE_DEV_LOGIN=true` on the server. Tests: 12 new vitest cases in `server/devLogin.test.ts` covering all four gates and the admin-gated API round-trip; full backend suite 101/101 passing.
- **v4.3.0 — Publish-to-dev-DNS + Ops request + email/notifications roadmap (docs).** Three new operational guides, no code changes. **`deployment/PUBLISH-TO-DEV-DNS.md`** — non-admin step-by-step to cut the dev build over from `http://DTIAPPSINTDEV:8080` to `https://cpactions-dev.cabq.gov`: Phase 0 verify prereqs, Phase 1 install `.pfx` via `certlm.msc`, Phase 2 add HTTPS binding with SNI in IIS Manager, Phase 3 URL Rewrite HTTPS canonical-host redirect, Phase 4 Entra redirect URI + logout URL, Phase 5 edit `.env` for `VITE_AZURE_REDIRECT_URI`, Phase 6 rebuild + redeploy via existing `scripts\push-to-sandbox.ps1` / `scripts\deploy.ps1`, Phase 7 smoke tests + rollback + troubleshooting (AADSTS50011/700038, 502, cert-authority-invalid, SPA 404). **`deployment/OPS-REQUEST-DNS-CERT.md`** — copy-paste Ops ticket covering internal DNS for the new hostname, TLS cert (SANs, delivery format, renewal contact, internal-CA trust, production-cert policy), firewall inbound TCP 443 from LAN + VPN, and SMTP relay info + dedicated `comp-plan-noreply@cabq.gov` sender + optional `comp-plan-support@cabq.gov` mailbox. **`docs/EMAIL-NOTIFICATIONS-ROADMAP.md`** — reserves env var names (`NOTIFICATIONS_ENABLED`, `SMTP_*`, `NOTIFICATIONS_ALLOWED_DOMAINS`), locks in the outbox-pattern architecture (queue -> drain worker -> channel adapter) so transient relay outages never lose a message, planned `server/notifications/` module layout + migration 6 schema (`notifications`, `notification_preferences`) with status lifecycle, hook points in existing code (`submissionsRepo`, `localAuthRoutes`, `auditRepo`), Handlebars template format, admin UI plan (Notifications tab under Security nav), day-one safety rules (domain allow-list, dev-redirect, per-user preferences, rate limits, audit coverage), and phased rollout v5.0.0 -> v5.3.0 mirroring the Reports initiative.
- **v4.2.1 — Lockfile sync (chore).** Pulled the sandbox server's post-`npm audit fix` `package-lock.json` back into the dev repo so future deploys don't regress the patched versions. All bumps are patch/minor within existing semver ranges — no application code changes. Notables: `axios` 1.14.0 → 1.15.1, `fastify` 5.8.4 → 5.8.5, `follow-redirects` 1.15.11 → 1.16.0, `postcss` 8.5.8 → 8.5.10, `vite` 8.0.3 → 8.0.9, `rolldown` rc.12 → rc.16 (all platform bindings), plus `@emnapi/*`, `@napi-rs/wasm-runtime`, `@oxc-project/types`, `tinyglobby`, and `nanoid` hoist. Validation: `npm audit` → 0 vulnerabilities, lint clean, full test suite 89/89 passing across 18 files, production build clean.
- **v4.2.0 — Reports, phase 3 of 3 (final).** Closes out the Reports initiative. **DB migration 5** adds `submission_status_history` (from/to_status, actor, at, FK-cascaded on submissions.id, indexed on (submission_id, at) and at); on first run it **backfills** one "created as draft" row per existing submission plus a `draft → submitted` row at `submitted_at` / fallback `updated_at` so the new report is immediately meaningful. `insertSubmission`, `patchSubmission` (user), and `patchAny` (admin) in `submissionsRepo.ts` now append a transition row whenever status effectively changes. **Report #4 — Submission Lifecycle / Turnaround** (`#reports/lifecycle`): KPIs (in-draft now, submitted total, median/p90 draft → submitted, median/max open-draft age with warning tone past 7d / 30d), two stats tables (n/min/median/p90/max/avg in auto-scaled human units — `Xm / Xh / Xd / Xmo`), top-15 oldest drafts linking into the admin detail page, and a by-month submissions + median-turnaround bar chart & table. New route: `GET /api/admin/reports/lifecycle`. Tests: 2 new cases; full suite 89/89 passing across 18 files. All five planned reports (Submissions Overview · User Activity · Authentication & Security · Coverage / Gap Analysis · Submission Lifecycle) are now live.
- **v4.1.0 — Reports, phase 2 of 3.** Adds **Authentication & Security** (`#reports/auth-security`) and **Coverage / Gap Analysis** (`#reports/coverage`). *Auth & Security* rolls up the `auth_audit` table into KPIs (logins ok/fail, lockouts, password/user/role/SSO changes), a daily stacked chart (7/30/60/90/180d), a top-15 failed-login watchlist with a **distinct-IP** column, and a latest-activity tail; exports to CSV via `GET /api/admin/reports/auth-audit.csv?days=1..365` using an authenticated `fetch`+`Blob` flow so identity headers travel. *Coverage / Gap Analysis* cross-references the comp-plan hierarchy JSON (`public/data/` or deployed `dist/data/`) against every submission's `planItems[]` and reports KPIs, a coverage-by-chapter bar table, the **uncovered goals list** (the gaps), and a saturation view of most-cited goals; uncovered-goals CSV is exported client-side. New routes: `GET /api/admin/reports/auth-security?days=7..180`, `GET /api/admin/reports/auth-audit.csv`, `GET /api/admin/reports/coverage`. Tests: 5 new cases; full backend suite 35/35. Next: **v4.2.0** adds Submission Lifecycle / Turnaround (needs migration 5 for status history).
- **v4.0.0 — Reports, phase 1 of 3.** Admin Console gains a new **Security** nav group (visual separator in the top nav containing Users · Roles · Sign-in settings · Audit log · **Reports**) and the first two reports. **Submissions Overview** (`#reports/submissions`) shows KPIs (total, submitted, draft, created/submitted last 7d + 30d), a 4/13/26/52-week stacked bar chart of submissions per week, and the top 10 most-cited comp-plan goals (chapter + goal names resolved from the hierarchy JSON). **User Activity** (`#reports/users`) lists every local user with roles, Active/Locked/Dormant badges, last-login days-since, and submitted/draft counts (filters: All · Active · Admins · Locked · Dormant > 90d); separately lists non-local submitters (SSO or header-based emails with submissions but no local account). Backend: `server/reports/reportsRepo.ts` + `server/reports/reportsRoutes.ts`; admin-gated `GET /api/admin/reports/submissions-overview?weeks=4..52` and `GET /api/admin/reports/user-activity`. Tests: 5 new vitest cases, full backend suite 30/30. Phase 2 (v4.1.0) will add **Authentication & Security** + **Coverage / Gap Analysis**. Phase 3 (v4.2.0) will add **Submission Lifecycle / Turnaround** (requires migration 5 for status history).
- **v3.8.3 — Deploy scripts.** New `scripts/deploy.ps1` (server-side) and `scripts/push-to-sandbox.ps1` (dev-box side). The server script does `npm install --omit=dev` (rebuilds `better-sqlite3` against the server's Node), verifies `tsx` is present, `pm2 reload`s with `--update-env --env production`, and probes `/api/health` + `/api/auth/config`. The dev-box script builds the SPA and robocopies `dist\`, `server\`, `package.json`, `package-lock.json`, `ecosystem.config.cjs`, and `scripts/deploy.ps1` to the mapped drive (default `Z:\cabq-plan`). `npm run deploy:sandbox` is the convenience shortcut. Together they prevent the `NODE_MODULE_VERSION` mismatch seen in v3.8.2.
- **v3.8.2 — Fix PM2 runtime on sandbox.** `tsx` moved from devDependencies → dependencies so `npm install --omit=dev` keeps it. `ecosystem.config.cjs` now launches `./node_modules/tsx/dist/cli.mjs` with `server/index.ts` (the previous `dist/index.js` entry never existed — `tsconfig.server.json` is `noEmit`). Resolves `ERR_MODULE_NOT_FOUND: Cannot find package 'tsx'` after a production reload.
- **v3.8.1 — Server `.env` auto-load.** `server/index.ts` now calls `dotenv/config` before any other import, so `LOCAL_JWT_SECRET`, `BOOTSTRAP_ADMIN_*`, `AZURE_*`, `ADMIN_*`, etc. can live in a single `.env` file next to the running Fastify process. Real environment variables (PM2 `env_production`, shell exports, etc.) still override the file. `.gitignore` now ignores `.env` / `.env.*` by default, with explicit allow-list entries for tracked templates.
- **v3.8.0 — Admin UI: tabbed sign-in + user / role / SSO management.** Admin console now offers a tabbed **Local / Microsoft (SSO)** login (driven by `GET /api/auth/config`), forces a password change on flagged accounts, and adds a header nav with **Submissions · Users · Roles · Sign-in settings · Audit log**. Users page lists local users with role checkboxes, admin-initiated password reset (forces change next login), deactivate, delete. Roles page creates/deletes custom roles (built-ins protected). Sign-in settings page edits tenant / client / audience / issuer / allowed domains / admin role names / admin emails, plus a live *Test SSO* dry-run against the server's JWKS. Audit log streams recent auth events with an action filter. Admin API client prefers the local-session JWT over MSAL silent token when both are present.
- **v3.7.0 — SSO managed from the admin.** Introduces an `auth_config` table and `getEffectiveAuthConfig()` helper that merges DB overrides over env defaults. New endpoints: public `GET /api/auth/config` (feeds the login UI); admin `GET/PATCH /api/admin/auth/config` for tenant id / client id / audience / issuer / allowed domains / admin role names / admin emails, plus `POST /api/admin/auth/test-sso` to dry-run a real Entra token. `resolveOwner()` and admin checks are now DB-aware (`isAdminFor(db, owner)`) so config changes take effect without restarting the server. Env vars remain the safe fallback / seed.
- **v3.6.0 — Local admin accounts (back-end).** New local-accounts table (`local_users`) with bcrypt-hashed passwords; short-lived HS256 JWTs (`LOCAL_JWT_SECRET`, 8 h default) feed straight into `resolveOwner()`, so every downstream endpoint accepts local sessions alongside Entra tokens. Admin CRUD for users and roles at `/api/admin/users` + `/api/admin/roles`; admin-initiated password reset (`/api/admin/users/:id/reset-password`) forces change on next login. Lockout after 5 bad attempts for 15 min; every auth/user/role change writes to `auth_audit` (visible at `/api/admin/auth/audit`). The last active `comp-plan-admin` cannot be deleted, deactivated, or demoted. First-run bootstrap admin is env-driven (`BOOTSTRAP_ADMIN_EMAIL` / `BOOTSTRAP_ADMIN_PASSWORD`).
- **v3.5.0 — UX polish.** Header shows **"Logged in as: {name}"** + **Sign out** + **Admin Console** (moved from footer). Save / Submit redirect to the **Submissions** tab (renamed from *Library*). Submitted records are now genuinely read-only (Tiptap + department combobox). *Reopen for editing* renamed to **Edit**. Composer action buttons are replicated at the **top** of the form. Server PDFKit fallback now mirrors the Print document layout so **Download PDF** and **Print document** produce the same output. Admin console **Print document** fixed (the hidden print layer is portalled outside `.admin-shell`).
- **Sprint 5 (v3.4.0) — Admin on the server.** New `/api/admin/submissions` endpoints list/patch all users' rows, guarded by **`ADMIN_ROLE_NAMES`** (default **`comp-plan-admin`**) **or** **`ADMIN_EMAILS`**. `admin.html` now requires MSAL sign-in + admin role, shows per-submission **`ownerEmail`** and status pills, and falls back to seeded localStorage with a notice when the API is unavailable. Auth context also carries **`roles`** (JWT claim or **`X-User-Roles`** header).
- **Sprint 4 (v3.3.0) — API JWT.** Server can validate **Entra access tokens** (`Authorization: Bearer`) using **`AZURE_TENANT_ID`** / **`AZURE_AUDIENCE`**; optional **`ALLOW_HEADER_IDENTITY=true`** keeps **`X-User-*`** trusted when migrating or before API scope consent. Client acquires **`VITE_API_SCOPE`** (default **`access_as_user`**) via MSAL silent token when configured.
- **Sprint 3 (v3.2.0) — Lifecycle.** Submissions carry **`draft` / `submitted`** status and **`submittedAt`**. API supports transitions and blocks deleting submitted rows. Composer: preview before submit, reopen, PDF download and email summary from the library.
- **Sprint 2 (v3.1.0) — Persistence.** Main app submissions live in **SQLite** via **`/api/submissions`** (Fastify). Routes: **`/app`** home (table), **`/app/compose`** composer. **`admin.html`** still uses localStorage until wired.
- **Sprint 1 (v3.0.0) — Entra ID + routes.** Public **landing** with **Sign in with Microsoft** (Azure Entra); **only @cabq.gov** (or `VITE_ALLOWED_EMAIL_DOMAINS`) may use the app after sign-in. See [`.env.example`](.env.example) and [`docs/VERSIONING.md`](docs/VERSIONING.md).
